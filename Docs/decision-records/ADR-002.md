# ADR-002: Use Repository Pattern

Status: Accepted

## Context

Business logic should not depend directly on EF Core.

## Decision

Introduce repositories as the abstraction between Application and Persistence.

## Alternatives

Inject DbContext directly

Pros

- Less code

Cons

- Tight coupling
- Difficult testing

Repository Pattern (Chosen)

Pros

- Testability
- Loose coupling
- Clear abstraction

Cons

- Extra abstraction

## Consequences

Repositories expose business operations rather than IQueryable.

## Interview Notes

Repository vs DbContext

Why not expose IQueryable?

## Why Not?

Why not DbContext?

Because Application would depend on EF Core.

Why not IQueryable?

Because it leaks persistence concerns.

Why not Services?

Because features become fragmented.
