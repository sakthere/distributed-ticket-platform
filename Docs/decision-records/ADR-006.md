# ADR-006: Introduce Result Pattern with Strongly Typed Errors

**Status:** Accepted

**Date:** 2026-07-14

---

# Context

Initially, the application represented business failures using plain strings.

Example:

```csharp
return Result<RegisterResponse>.Failure("Email already exists.");
```

While this approach was simple, it introduced several long-term issues as the application grew.

Problems identified:

- Magic strings scattered across handlers.
- Duplicate error messages.
- No unique identifier for failures.
- Difficult to standardize API responses.
- Difficult to localize messages.
- High risk of typos and inconsistent wording.
- Controllers had no reliable way to map business failures to HTTP status codes.

As more features are added, these problems would multiply and make the application harder to maintain.

---

# Decision

Introduce a standardized Result Pattern consisting of:

- `Error`
- `Result`
- `Result<T>`

Business failures are represented by strongly typed `Error` objects rather than raw strings.

Example:

```csharp
return Result<LoginResponse>.Failure(
    AuthErrors.InvalidCredentials);
```

Authentication-specific errors are grouped together in a dedicated `AuthErrors` class.

```text
Application
│
├── Common
│   ├── Error.cs
│   ├── Result.cs
│   └── ResultT.cs
│
└── Features
    └── Authentication
        └── AuthErrors.cs
```

This keeps ownership close to the feature that defines the business rule.

---

# Alternatives Considered

## Option 1 - Continue Using Strings

Example:

```csharp
return Result.Failure("Invalid credentials");
```

### Advantages

- Very simple.
- No additional classes.

### Disadvantages

- Magic strings.
- Duplicate messages.
- No error codes.
- No compiler support.
- Difficult localization.
- Difficult HTTP mapping.

**Decision:** Rejected.

---

## Option 2 - Throw Exceptions for Business Failures

Example:

```csharp
throw new InvalidCredentialsException();
```

### Advantages

- Centralized handling.
- Familiar pattern.

### Disadvantages

- Business failures are expected outcomes, not exceptional situations.
- Exceptions are expensive compared to normal control flow.
- Harder to distinguish expected failures from unexpected system failures.

**Decision:** Rejected.

---

## Option 3 - Use a Third-Party Library (FluentResults / ErrorOr)

### Advantages

- Mature implementations.
- Rich feature set.
- Community support.

### Disadvantages

- Hides important implementation details during learning.
- Less opportunity to understand trade-offs.
- Additional dependency.

**Decision:** Deferred.

The project intentionally implements its own lightweight Result Pattern first to understand the underlying design. Third-party libraries will be evaluated later.

---

# Consequences

## Positive

- Strongly typed business failures.
- Single source of truth for error definitions.
- Error codes can be shared across backend, frontend, logs, and support.
- Controllers no longer rely on string comparisons.
- Easier API standardization.
- Easier localization.
- Compiler-assisted refactoring.
- Better separation between business failures and unexpected exceptions.

## Negative

- Additional classes to maintain.
- Slight increase in initial complexity.
- Requires every feature to define its own error catalogue.

# Design Principles Applied

- Single Responsibility Principle (SRP)
- Dependency Inversion Principle (DIP)
- Don't Repeat Yourself (DRY)
- Feature Ownership
- Separation of Concerns

# Production Considerations

Business failures should not be represented by exceptions.

Unexpected failures (database outages, null references, infrastructure failures) should still throw exceptions and be handled by the Global Exception Middleware.

Expected failures (invalid credentials, duplicate email, validation failures) should return `Result<T>`.

This separation provides:

- Better performance.
- Cleaner code.
- Consistent API responses.
- Clear distinction between business and system failures.

---

# Interview Notes

### Why use a Result Pattern?

Business failures are expected outcomes and should not be represented as exceptions. A Result Pattern makes success and failure explicit, improves readability, and avoids using exceptions for normal control flow.

---

### Why not return strings?

Strings have no identity, cannot be standardized, are difficult to localize, and make HTTP response mapping unreliable.

---

### Why define feature-specific error classes?

Errors belong to the feature that owns the business rules. Authentication errors change with the Authentication feature, Ticket errors change with the Ticket feature. Grouping them together improves cohesion and follows the Common Closure Principle.

---

### When should exceptions still be used?

Exceptions should represent unexpected system failures such as infrastructure outages, programming errors, or external dependency failures. Business rule violations should be returned as `Result<T>`.

# Future Improvements

- Introduce centralized Result → HTTP mapping.
- Add standardized error codes for every feature.
- Evaluate adoption of FluentResults or ErrorOr once the team understands the underlying design.
- Add localization support using error codes.
- Integrate structured logging with error codes and correlation IDs.
